# AKV 行为测试报告

- 日期：2026-07-16
- 变更前基线：`9b80ea9`
- 结论：核心授权安全行为与四项试点指标的产品采集链 `PASS`；测试已生成并核对观测数据，但真实试点基线仍须由实际用户操作产生，且未预设改善幅度；发现并修复 1 个任务终态审批缺陷，另有 7 项后续风险。

## 测试范围

本轮把此前分散的单元测试串成两个参与者视角的真实行为旅程：

1. 普通用户通过 Web 注册并注册 Agent；Agent 发现安全操作、开始任务、发送心跳、申请授权；人类使用 Session Cookie 和 CSRF 批准；Agent 仅执行一次并读取关联审计。
2. Agent 在等待审批时结束任务；人类随后尝试批准，系统必须原子拒绝且不得创建 Approval 或 Grant。
3. 人类在待审批阶段显式记录两次人工转交和一次审批补问；执行完成后开始并完成同一复盘会话；系统从 PostgreSQL 服务端时间聚合四项指标。
4. 以固定合成向量继续验证统计公式，同时覆盖历史未知、新申请明确零值、幂等重试、跨用户拒绝、追加不可变和复盘并发完成。

系统测试使用临时 PostgreSQL、真实 control/execution HTTP handler、进程内受保护目标和内存 Vault fake。没有连接真实 OpenBao、真实目标系统或真实凭证。

## 试点观测指标

“申请到结果”在本报告中明确指执行结果完成，不与审批决定时间或回收完成时间混用。第一项使用既有授权与执行时间边界；另外三项现由独立的追加式观测表采集，不能从审计 actor 切换、重复申请或 HTTP 服务端耗时猜测。

| 指标 | 统计口径 / 数据源 | 本轮验证 | 真实试点值 | 改善幅度 |
| --- | --- | --- | --- | --- |
| 申请到结果用时 | `executions.completed_at - authorization_requests.created_at` | PASS：真实临时 PostgreSQL 完成链区间存在且非负 | 待真实试点 | 未预设 |
| 人工转交次数 | `MANUAL_HANDOFF` 追加事件计数 | PASS：Web 显式记录 2 次，数据库汇总为 2；同键重试不重复 | 待真实试点 | 未预设 |
| 审批补问次数 | `APPROVAL_FOLLOWUP` 追加事件计数 | PASS：Web 显式记录 1 次，数据库汇总为 1 | 待真实试点 | 未预设 |
| 复盘一次操作所需时间 | 同一复盘会话 `REVIEW_COMPLETED - REVIEW_STARTED` | PASS：执行前拒绝；执行后配对并由服务端时间得到非负区间 | 待真实试点 | 未预设 |

迁移前的历史申请没有采集启用记录，三项人工指标返回 `null/UNKNOWN`；迁移后创建的新申请从开始即启用采集，未记录事件时计数返回明确的 0。人工转交和审批补问只允许在申请待审批且未过期时记录，复盘只允许在执行结果完成后开始，并要求同一申请、同一发起人完成。

观测 API 只接受空 JSON 与幂等键，不接受自由文本、actor、时间或凭证字段；时间统一由 PostgreSQL 分配，事件不可更新或删除。观测表不参与审批、Grant 占用、执行或回收判断，因此没有改变授权安全语义。

## 发现并修复

### F-001：已结束任务仍可创建 Grant

初次运行新回归时，Agent 将任务结束为 `CANCELLED` 后，人类审批仍返回 HTTP 200，数据库产生 1 条 Approval 和 1 条 Grant。执行守卫仍会因任务非活动而拒绝 Claim，因此未造成凭证访问，但会形成不可执行的僵尸 Grant 与错误审计记录。

修复在 PostgreSQL 的原子审批条件中加入任务所有权和 `ACTIVE` 状态校验，并在批准事务内锁定活动任务行，使并发任务结束必须在 Grant 创建后完成回收。修复后相同旅程返回 HTTP 409，且 Approval、Grant、Vault 读取和目标调用均为 0。

## 后续风险

以下问题没有阻断本轮核心授权行为验收，但应进入后续测试与修复队列：

1. `中`：Agent 注册和 Token 轮换界面目前固定为 24 小时，未提供架构定义的 30 天和永久选项及永久风险提示。
2. `中`：Web 退出请求失败时仍切回登录页，刷新后可能因服务端 Session 尚有效而恢复登录，界面会产生“已安全退出”的错觉。
3. `中`：多个进程并发执行数据库迁移时没有 PostgreSQL advisory lock，空库首次启动可能发生 DDL 竞争。
4. `中`：全局审计和申请审计页面未展示 approval、grant、execution、reclaim 的全部关联 ID。
5. `低`：审批竞争、撤销失败和目录失败仍可能直接显示内部错误码，缺少面向人类的可行动提示。
6. `高`：旧 PostgreSQL 测试在脱离官方临时数据库脚本单独运行时仍直接信任测试 DSN；新行为测试已增加强门禁，但旧测试尚未统一迁移到安全 fixture。
7. `中`：人工转交和审批补问依赖用户主动点击；漏记会导致低估。试点执行时应培训参与者并核对采集覆盖率，不能把“没有点击”解释为对现实过程的独立证明。
