# AKV Agent 开发规则

目标：依据 `docs/project-requirements.md` 和 `docs/architecture.md` 自主实现 AKV MVP；无真实阻塞时持续完成“实现—验证—记录—本地提交”循环。

## 读取与优先级

优先级：当前用户指令 > 需求文档 > 架构文档 > 本文件 > `.agent/memory.md` > `.agent/progress.md`。

每次恢复只需读取：

1. `.agent/progress.md` 当前任务；
2. `git status` 和相关代码；
3. 架构文档中与当前任务相关的章节；
4. 需要确认长期决策时再查 `.agent/memory.md`。

不要每轮重复加载、复述整份需求和架构。遇到范围、安全或验收歧义时再读取完整需求。需求文档默认只读；不得静默改变架构安全语义。

## 安全红线

- Agent、Web、Prompt、日志和错误不得得到目标源凭证明文。
- 必须先验证 Agent/任务/上下文并在数据库中原子占用 Grant，之后才能访问 OpenBao 或目标系统。
- Grant 强绑定 Agent、任务、目标、凭证和操作快照；只能执行一次，失败或重试也需重新审批。
- 未审批、拒绝、过期、撤销、重放或上下文不匹配时必须默认拒绝，且不得调用 OpenBao 或目标系统。
- 所有终态都要回收；静态源凭证保留，临时凭证、Lease 和会话撤销销毁；回收失败时保持阻断并告警。
- 动态 PostgreSQL 凭证签发失败不得降级为固定凭证。
- 不使用真实目标凭证做开发或测试，不把目标凭证写入文件、环境变量、命令、测试数据或 `.agent` 文档。本地 MVP 允许把 AKV Agent 身份 Token 持久化在项目根目录 `.agent-token`，但该文件必须被 Git 忽略且权限为 `0600`；Token 不得输出、提交、复制到其他文件或写入 Prompt、命令参数、日志、测试数据和 `.agent` 文档。

授权、执行和回收功能至少测试：未审批、并发占用、重放、跨 Agent、跨任务、过期、撤销、失败和回收失败。

## 自主循环

每轮只处理一个最小、可验证、可提交的工作项：

1. 对照代码校正 progress；保留用户已有改动。
2. 选择第一个依赖已满足的 `READY` 项，标为 `IN_PROGRESS`，写明验收条件。
3. 实现最小改动并补测试；状态并发安全必须落在持久层，外部系统使用接口和 fake 隔离。
4. 运行针对性测试、完整验证、格式/静态检查和 `git diff --check`。
5. 自审 diff，确认没有越界改动、敏感数据或调试残留。
6. 更新 progress；只有产生长期有效事实时才更新 memory。
7. 显式暂存本轮文件、检查 staged diff、创建本地提交，然后继续下一项。

统一入口建立后使用 `make verify`。验证不能运行或失败时不得标记 `DONE`、不得提交成功状态。

## 决策与阻塞

可自主决定可逆的内部实现、测试结构和最小依赖，并把稳定结论写入 memory。以下情况必须在 progress 记录证据并询问用户：

- 修改需求、角色、审批语义、凭证流向或安全边界；
- 破坏性迁移、真实外部系统操作或真实凭证处理；
- 引入昂贵且难替换的基础设施；
- 选择会导致明显不同产品行为，或安全替代方案均失败。

## Git 规则

- 用户授权完成工作项后创建本地提交，但未授权自动 push、发布或部署。
- 若尚无 Git，首次循环可 `git init` 并创建安全 `.gitignore`；不得自动添加远程。
- 缺少 Git 身份时可仅为本仓库设置 `AKV Development Agent <akv-agent@local.invalid>`。
- 只用显式路径暂存当前工作项；禁止 `git add .`、`git add -A`、`--no-verify`。
- 使用小型 Conventional Commit，并带 `Task: AKV-NNN`；代码、测试和本轮 agent 文档一起提交。
- 禁止自动 push、rebase、amend、reset --hard、checkout --、clean 或覆盖用户改动。

`DONE` 表示：验收满足、相关测试及完整验证通过、progress/memory 已更新且本地提交成功。

## 文档节制

- progress 只保留当前任务、短队列、最新验证和最近 10 轮日志。
- memory 只记稳定事实，不复制需求/架构，不写 TODO 或流水账。
- 输出和提交说明保持简短；能链接代码或文档时不重复粘贴内容。
